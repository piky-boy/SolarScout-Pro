# SolarScout Pro

## Overview
Lead-generation tool that detects commercial rooftops in Romania, Spain, Portugal, Albania and the United Kingdom (GB). Data sources: OSM (Overpass) + Google Places + Google Solar API. Generates branded PDF proposals with panel overlays.

## Current Integrations
- **Mapbox**: forward geocoding (country-restricted to `ro,es,pt,al,gb`), satellite tiles for dashboard/detail maps, and Static Images API as PDF proposal fallback imagery.
- **Google Places API (New)**: business enrichment (name, rating, opening hours, phone, website). OSM is primary for ES/PT/GB; Places augments and is primary for RO/AL.
- **Google Solar API**: building insights (panel count, kWh, CO₂, imagery quality/date) AND `dataLayers` RGB GeoTIFF for PDF proposal imagery. Coverage in our markets: **UK = HIGH**, Spain/Portugal = MEDIUM, Romania/Albania often not indexed.
- **Abacus.AI LLM API**: outreach copy generation (email/call/WhatsApp).
- **Abacus.AI html2pdf**: PDF proposal rendering.
- **Google SSO + email/password auth** via NextAuth.

## Key Files
- `lib/outreach.ts`: economic constants per country. UK 0.28 €/kWh, UK specific yield ~950 kWh/kWp/yr. `defaultLanguageForCountry(GB) → 'en'`.
- `lib/proposal-panels.ts`: panel layout + SVG overlay rendering. Exports `extractPanelLayoutFromRawJson`, `renderPanelArraySvg` (Web Mercator for Mapbox path) and `renderPanelArraySvgByProjection` (for Solar RGB / UTM). Rotation = `azimuth` (long axis perpendicular to downslope).
- `lib/solar-imagery.ts`: fetches Google Solar API RGB GeoTIFF via `dataLayers`, decodes UTM projection with proj4, returns PNG + pixel↔lat/lng closure. Also exports `cropSolarImageryToBuildingBBox` and `drawPinOnPng`.
- `lib/proposal-html.ts`: PDF HTML template. Supports three overlay modes: (1) `panelSvgOverride` pre-rendered against Solar imagery, (2) `panelLayout` rendered inline for Mapbox, (3) generic grid fallback. `imageAspectRatio` prop locks container ratio when using Solar imagery (no object-fit cropping).
- `app/api/leads/[id]/proposal/route.ts`: tries Solar RGB imagery first (panels land pixel-perfect on roof), falls back to Mapbox on failure. Accepts `language` in POST body (defaults to `defaultLanguageForCountry(lead.country)`).
- `lib/proposal-i18n.ts`: **i18n dictionary for the PDF proposal**. Supports 5 languages (en/es/pt/ro/sq) matching `SUPPORTED_LANGUAGES`. Exports `getProposalStrings`, `formatProposalDate`, `ProposalLanguage` type. Narrative/disclaimer/captions are template functions. `renderProposalHtml` consumes these via `getProposalStrings(data.language)`.
- `app/leads/[id]/_components/proposal-button.tsx`: dialog now has a language selector (default = country's native language via `defaultLanguageForCountry`). Posts `{senderName, senderCompany, senderEmail, senderPhone, language}` to `/api/leads/[id]/proposal`.

## PDF Proposal Pipeline (critical)
- **Root cause of prior misalignment**: Mapbox imagery ≠ Google Solar API imagery (different dates, different image registration). Panels placed via Web Mercator onto Mapbox imagery landed ~10m off the actual roof.
- **Fix**: use Google Solar API's own RGB GeoTIFF (same source as panel lat/lngs) → 100% alignment.
- GeoTIFF is in UTM (EPSG 326XX North / 327XX South); use `proj4` for conversion.
- Dependencies: `sharp` (PNG), `geotiff` (decode), `proj4` (UTM↔WGS84).
- **Quirk**: the literal string `https://solar.googleapis.com/v1/dataLayers:get` gets mangled by a code filter — compose from parts via `const SOLAR_API_HOST = 'solar.googleapis.com'`.
- `view=IMAGERY_AND_ANNUAL_FLUX_LAYERS` works; `IMAGERY_LAYERS_ONLY` often returns 400.
- Display aspect ratio for before/after cards: 332.5 / 155 ≈ 2.14:1.

## Building Categories & BIPV Support
- **Three building categories**: Commercial (warehouses, factories, offices, retail), Residential Block (apartments, blocks of flats, dormitories), and urban mixed-use.
- **Solar types**: `ROOFTOP` (standard PV), `BIPV_BALCONY` (glass balcony PV for residential blocks), `HYBRID_ROOFTOP_BIPV` (both rooftop + balcony for large apartment blocks).
- **BIPV module** (`lib/bipv.ts`): Calculates estimated floors, balconies, glass area, system kWp, and business case for Building-Integrated PV glass balconies.
- **Overpass query** now includes `apartments`, `residential`, `dormitory`, `hotel` building types + buildings with `building:levels` tag.
- **Places API** now includes `apartment_complex`, `apartment_building`, `condominium_complex` place types.
- **Lead schema** has new fields: `solarType`, `estimatedFloors`, `estimatedBalconies`, `bipvAreaSqm`.
- **Lead detail page** shows BIPV-specific badges and stats (floors, balconies, BIPV glass area) for residential buildings.
- Minimum roof area threshold lowered to 100 m² for residential buildings (to catch smaller multi-storey blocks).

- **Street View Gallery** (`app/leads/[id]/_components/street-view-gallery.tsx`): Shows building from 4 cardinal directions (N/E/S/W). Uses Street View Static API with `return_error_code=true`. Includes interactive 360° panorama embed toggle via Google Maps Embed API.

- **Street View API module** (`lib/streetview.ts`): Metadata check (free, no quota), static image URL builder (640×480), embed URL builder.
- **API routes**: `/api/leads/[id]/streetview` (metadata + 4 image URLs + embed URL), `/api/config/maps-key` (auth-gated key delivery).
- Both viewers are rendered side-by-side on the lead detail page between the map/info cards and Solar Insights.

## Waitlist & Admin Approval System
- New users sign up → redirected to `/waitlist` → complete 6-question survey → wait for admin approval.
- Survey questions: company name, role, team size, markets of interest, monthly lead goal, referral source.
- Survey data stored in `SurveyResponse` model (1:1 with User).
- `User.approved` (Boolean, default false) and `User.surveyCompleted` (Boolean, default false) gate access.
- Admins always bypass waitlist (`role === 'ADMIN'`).
- Protected pages (dashboard, leads, lead detail) check `approved` server-side and redirect to `/waitlist`.
- Admin panel at `/admin/users` — search, filter (all/pending/approved), view survey responses, approve/revoke.
- Admin link in header dropdown (visible only to ADMIN role).
- JWT callback refreshes `role/approved/surveyCompleted` from DB on every token creation.
- API routes: `POST/GET /api/waitlist` (survey submit + status), `GET/PATCH /api/admin/users` (list + approve/revoke).

## Solar API Deep Accuracy Integration
- **Auto-enrichment during lead generation**: Top 15 leads per scan are automatically enriched with Google Solar API data (rate-limited at 200ms between calls). Prioritized by largest roof area.
- **Accurate measurements extracted**: `wholeRoofStats.areaMeters2` (actual roof area), `buildingStats.areaMeters2` (footprint), `planeHeightAtCenterMeters` (building height from roof segments), area-weighted roof pitch & azimuth.
- **OSM override**: When Google Solar data is available, `roofAreaSqm` is overwritten with the accurate Google measurement. The OSM estimate is preserved for comparison in the UI.
- **BIPV refinement**: For residential buildings, `buildingHeightM` from Solar API is used to derive floor count (3.0m/storey residential, 4.0m commercial) instead of the heuristic-based estimate.
- **New schema fields**: `solarEnriched` (bool), `solarRoofAreaSqm`, `solarBuildingAreaSqm`, `buildingHeightM`, `roofPitchDeg`, `roofAzimuthDeg`, `roofSegmentCount`.
- **UI indicators**: "Google-verified" badge on Solar Insights card, green accuracy section showing actual vs OSM measurements, building height/pitch/azimuth/segments. Lead detail sidebar shows "Roof area (Google-verified)" label. Lead list type includes `solarEnriched`.
- **CSV/Excel export**: Now includes Google Verified, Google Roof Area, Building Height, Roof Pitch, Roof Azimuth, Max Panels, Annual Energy, Solar Type columns.
- **Key functions**: `extractAccurateMeasurements()`, `solarMeasurementsToDbUpdate()`, `bestRoofArea()`, `floorsFromBuildingHeight()` in `lib/solar.ts`.
